## 슬라이드 1 — 표지 + 프로젝트 1페이지 요약

**제목:** 편의점 매대 정리 로봇 — 시뮬레이션 파이프라인 검증
**부제:** GraspGen(AI 파지 추론) → cuRobo(충돌회피 모션플래닝) → Isaac Sim(물리 기반 검증)
**날짜:** 2026-06-11 | **발표자:** 남정혁

### 본문 (1페이지 요약 박스 3개)

**📦 GraspGen — AI 파지 후보 생성**
- 물체 점구름(PointCloud) 입력 → 파지 자세(pose) 다수 후보 출력
- 백본: PointNet++ (Blackwell/CUDA 12.8 호환)
- ZMQ 서버 분리 → Isaac Sim 프로세스와 디커플
- 물체 3종: 스낵백 / 캔 / 페트병

**🦾 cuRobo — 충돌회피 모션플래닝**
- 목표 pose → 충돌 없는 관절 궤적 생성
- 충돌구체(Collision Sphere) 모델로 로봇 기하 표현
- plan_single / plan_grasp(goalset) v1 API (cuRobo v0.7.8)
- 환경: 책상 + 매대 + 이웃 물체를 실시간 장애물 등록

**🔬 Isaac Sim — 물리 기반 검증**
- E0509 6축 암 + RH-P12-RN-A 그리퍼 USD 실기체 모델
- 물리엔진(PhysX)으로 질량·마찰·접촉 시뮬레이션
- PD 게인(arm kp800/kd80, gripper kp600/kd100) 실기체 동일 적용
- 결과: 파지 → 리프트 → 매대 직립 적치 → 홈복귀 전 구간 검증

**이미지 배치:**  
→ 파이프라인 플로우 다이어그램 (GraspGen→cuRobo→Isaac Sim 화살표)

## 슬라이드 4 — 팀 공유 2: 관절 제한 범위 공유

**제목:** 팀 공유 — 관절 자세 제한값 (실기 적용 예정)

### 문제: 왜 제한이 필요했나

시뮬에서 발견: cuRobo가 운반 중 **손목 플립(j5 부호 반전)** 선택  
→ j4가 180° 이상 돌며 이웃 물체·매대와 간섭 위험

```
[문제 상황 관측 데이터]
파지   : j4 = +143°, j5 = +81°  (손목 위 ← 좋음)
운반   : j4 = +269°, j5 = −92°  (손목 아래 ← 나쁨, 플립 발생)
홈복귀 : j4 204° 되감기          (↑ 이웃 물체 통과 위험)
```

### 해결: 비대칭 관절 제한 (URDF 적용)

| 관절 | 기존 범위 | 수정 범위 | 이유 |
|------|-----------|-----------|------|
| joint_4 (팔꿈치) | −360° ~ +360° | **−180° ~ +180°** | 과회전/홈 되감기 차단 |
| joint_5 (손목 롤) | −135° ~ +135° | **0° ~ +135°** | 손목-하향(플립) 분기 차단 |

**검증 결과 (오늘 시뮬):**

```
파지   : j4 = +143°, j5 = +81°   ✅ 유지
운반   : j4 = +90°,  j5 = +92°   ✅ 손목 위 유지
적치   : j4 = +90°,  j5 = +63°   ✅ 
홈복귀 : j4 = +65°,  j5 = +115°  ✅ 플립 없음
```

**→ 실기 팀에 이 범위 적용 권장. 도달 불가 케이스 없음(3방향 모두 파지 성공 확인).**

## 슬라이드 5 — 문제해결 1: 충돌구체 커스터마이징

**제목:** 충돌구체 재생성 — "cuRobo가 실제 로봇을 보게 만들기"

### 배경: 충돌구체란

cuRobo는 로봇 링크를 **구체 집합**으로 근사해 장애물 거리를 계산함  
구체가 틀리면 → **실제 충돌인데 통과** 또는 **멀쩡한데 거부** 두 가지 오류

### 기존 yaml의 문제 (발견 경위)

- 팔이 책상에 실제로 닿는데 cuRobo가 "충돌 없음" 통과
- 구체 시각화로 확인 → **link_2 구체가 링크 본체를 전혀 덮지 않음**

```
link_2 실제 모양: X축으로 490mm 긴 링크 (팔의 가장 긴 세그먼트)
기존 yaml:       구체가 모두 z축 (x=0, y=0) — 링크 90% 비어 있음
→ cuRobo가 link_2를 충돌 검사에서 못 봄
```

### 해결: cuRobo 공식 API로 메시에서 직접 피팅

```python
# cuRobo 공식: curobo.geom.sphere_fit.fit_spheres_to_mesh
spheres = fit_spheres_to_mesh(
    mesh_file,
    n_spheres=10,          # 링크별 개수 조정
    method=SphereFitType.VOXEL_VOLUME_SAMPLE_SURFACE
)
```

- 링크 URDF 메시(STL) → 복셀화 → 구체 자동 배치
- 그리퍼: 표면 샘플 (손가락 사이 공간 유지 → 파지 가능)
- base_link: 책상 마운트 항상 겹침 → 충돌 목록 제외

### 결과

| | 이전 | 이후 |
|--|------|------|
| link_2 coverage | ~10% | ~95% |
| 책상 충돌 감지 | 통과(오류) | 정확 거부 |
| plan_single 성공률 | 낮음 | 안정적 |

**이미지:**  
→ 왼쪽: 기존 구체 (z축 치우침) vs 오른쪽: 새 구체 (링크 덮음)  
→ shot_004_grasp_reached.png 또는 shot_006 (구체가 로봇에 밀착된 모습)

## 슬라이드 6 — 문제해결 2: 룰베이스 → 플래너 선택

**제목:** 파지점 선택: 하드코딩 규칙 → 플래너 자동 선택

### 기존 방식: 룰베이스 (문제)

```
GraspGen 후보 50개 → 수동 가중치 필터 
  (손목 부하 + 정면 페널티 + 클리어런스 점수)
→ 1위 후보로 접근 시도
→ 클러터 위치 바뀔 때마다 가중치 재튜닝 (하루 3회)
```

**문제점:**
- 클러터를 가중치 계산에 반영 안 함 → 접근 중 이웃 캔 충돌
- 후보가 막혀도 같은 후보 재선택 → 무한루프
- "장면이 바뀌면 망가지는" 취약한 선택

### 해결: plan_grasp(goalset) — 플래너가 선택

```python
# 후보 N개를 Pose(1, N, 7) 묶음으로 cuRobo에 전달
gres = motion_gen.plan_grasp(
    cu_js,          # 현재 관절 상태
    gposes,         # (1, N, 7) 전체 후보
    plan_config,
    grasp_approach_offset = ...,
    disable_collision_links = GRIPPER_COLL_LINKS
)
best_idx = gres.goalset_index  # planner가 고른 최적 후보
```

- cuRobo가 현재 월드(이웃 캔 포함)를 보면서 충돌 없는 접근 가능한 후보 자동 선택
- 룰베이스 가중치·정면 페널티 전부 삭제

### 검증 결과

| 케이스 | 기존 | 변경 후 |
|--------|------|---------|
| 클러터 0개 | 성공 | 성공 |
| 클러터 1개 (+y) | 종종 충돌 | goalset→−y 접근 자동 선택 |
| 클러터 2개 (양옆) | 무한루프 | goalset→정면 자동 선택 ✅ |
| 접근 중 캔 이동량 | ~0.4m (넘어짐) | **~5mm** ✅ |

**이미지:**  
→ shot_001_grasps.png: 파지 후보 (여러 방향) → 그 중 RGB 축이 실제 선택 후보  
→ shot_004_grasp_reached.png: 클러터 사이 진입 성공

## 슬라이드 7 — 문제해결 3: 불합리한 모션 자세 수정

**제목:** 운반 중 팔이 뒤집히는 문제 해결

### 관찰된 문제 (왼쪽 이미지)

> "파지 후 매대로 이동하면서 팔이 y−쪽으로 크게 휘었다가 오고, 매대 진입 시 link4·5가 아래를 봄"

- **원인:** cuRobo는 자세(posture) 비용이 없음. 목표 pose를 가장 짧은 경로로 풀면서 IK 분기가 바뀜
- 파지 시 손목 위(j5=+81°) → 운반 중 손목 아래(j5=−92°)로 분기 이주
- j4가 269°→ 홈복귀 시 204° 되감기 → 이웃 물체 통과 위험

### 해결 (오른쪽 이미지)

비대칭 관절 제한을 **URDF에서** 적용 (런타임이 아닌 생성 시점)

```python
make_jlim_urdf(원본_urdf, 수정본_urdf, {
    "joint_4": (-180°, +180°),   # 과회전 차단
    "joint_5": (  0°, +135°),   # 손목-하향(플립) 분기 완전 차단
})
# MotionGen 생성 전에 수정본 경로 사용
robot_cfg["kinematics"]["urdf_path"] = 수정본_urdf
```

**핵심:** BoundCost가 URDF를 초기화 시점에 클론 → 런타임 수정 무효. URDF 파일로 줘야 반영됨.

### Before / After

| 상태 | Before (j5) | After (j5) |
|------|------------|------------|
| 파지 | +81° ✅ | +81° ✅ |
| 운반 | **−92° ❌ (플립)** | **+92° ✅** |
| 적치 | 불안정 | +63° ✅ |
| 홈복귀 j4 되감기 | **204°** | **25°** ✅ |

**이미지:**  
→ 왼쪽: 이전 "팔 아래 향함" 스크린샷 (사용자가 전달했던 이미지)  
→ 오른쪽: shot_006_carry_preshelf.png (팔꿈치 위, 깔끔한 자세)

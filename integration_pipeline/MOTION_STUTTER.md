# 실로봇 모션 끊김(뚝뚝) — 원인과 수정안

## 증상 (정확히)

**구간(세그먼트) 수 문제가 아님.** **하나의 경로 계획 후, 그 단일 궤적을 실행하는 도중** 로봇이
"가다 → 멈칫 → 뒤로 갔다 → 다시" 하는 식으로 출렁임. (예: 홈→파지점 한 번의 plan 실행 중.)

## 진단 (핵심)

- **궤적(plan) 자체는 정상.** 시뮬에서 동일 궤적이 매끄럽게 실행됨 → cuRobo 계획·관절한계 문제 아님.
- 문제는 **그 궤적을 실로봇 컨트롤러로 전송/추종하는 단계.**
- ★**"뒤로 갔다(reverse)"** 가 결정적 단서: 매끈한 궤적은 역주행할 이유가 없음 → **컨트롤러가 목표를 지나쳤다 되돌아오거나(overshoot), 명령이 불규칙/역순으로 도착**한다는 뜻.

### 유력 원인(순위)

1. **궤적을 점(waypoint)으로 잘게 스트리밍** — cuRobo 궤적을 servoj/위치 setpoint로 파이썬 루프에서 하나씩 전송. 파이썬+ROS2+통신으로 **전송 간격 들쭉날쭉** → 서보가 한 점 가다 다음 명령 지연 시 도달/오버슛 → 미세 역주행 누적.
2. **속도 없이 위치만 전송** — cuRobo 궤적엔 pos+vel+time 있음. 위치만 주면 컨트롤러 PID가 매 점 가감속/오버슛 → 출렁.
3. **통신 지터/지연** — 스트리밍이면 지터가 그대로 끊김/역주행.
4. **레이트 불일치** — cuRobo `interpolation_dt=0.03`(33Hz) vs 두산 서보 주기 불일치 → 점 사이 출렁.
5. (가능) 컨트롤러 게인 과강성 / 속도 피드포워드 미사용 → 오버슛 진동.

## 수정안

- **A (권장): 단일 궤적을 통째로 시간궤적 인터페이스에 전달.**
  ROS2 `FollowJointTrajectory`(ros2_control) 또는 두산 trajectory/`movesj`(pos+vel+time)로 한 번에 보내고 **컨트롤러가 내부 보간**으로 실행. **파이썬 점-스트리밍 금지.**
  - cuRobo 결과: `result.get_interpolated_plan()` → `motion_gen.get_full_js(...)` 에 position·velocity·시간(dt)이 있음. 이를 trajectory 메시지로 변환해 한 번에 전송.
- **B (servoj 스트리밍이 불가피하면):**
  - **고정 실시간 주기**로 전송(두산 서보 주기에 맞춤, 예 8ms). 파이썬 일반 루프 금지(실시간 스레드/버퍼링).
  - **속도 포함** 전송(pos+vel).
  - cuRobo `interpolation_dt`를 서보 주기에 맞춰 **재보간**(33Hz→서보 레이트).
  - 미리 N스텝 **버퍼링**, 궤적 도중 **재전송/재계획 금지**.
- **C: 통신·게인 점검.** QoS reliable·저지터, 컨트롤러 게인·속도 피드포워드 확인.

## 체크 순서(빠른 진단)

1. 지금 궤적을 **어떻게 보내는지** 확인 — 점 단위(servoj/위치 setpoint 반복)인가, 통째(trajectory action/movesj)인가?
2. 점 단위면 → **A로 전환**(통째 trajectory)부터 시도. 대개 이걸로 해결.
3. 그래도 남으면 → 전송 주기/속도포함/버퍼링(B) + 통신·게인(C).

> 요약: **계획이 아니라 "단일 궤적의 실시간 전송/추종"** 이 원인. 점-스트리밍을 통째-trajectory로 바꾸는 게 1순위.

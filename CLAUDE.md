## 작업 규칙
1) 코드 다수 수정 필요 시 먼저 묻고 진행. 임의 실행 금지.
2) 기능별 모듈화: perception_bridge / grasp_inference / motion_planning / sim_validation
3) 요청 불명확하면 추론·실행 전에 이해한 바를 먼저 확인.
4) 보고 → 승인 → 진행. 각 단계 독립 검증 후 다음. (Plan Mode 사용)

  stop-sentinel(`touch /tmp/stage4_stop`)도 graceful 종료로 여전히 지원하나, 셧다운이 길어지면 kill -9 사용.

# Claude 작업 지침 (행동 가이드라인)

흔한 LLM 코딩 실수를 줄이기 위한 지침. 위 프로젝트 고유 규칙과 함께 적용한다.

**트레이드오프:** 이 지침은 속도보다 신중함에 무게를 둔다. 사소한 작업은 판단에 맡긴다.

## 1. 코딩 전에 생각하기

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현 전에:
- 가정을 명시한다. 불확실하면 묻는다.
- 해석이 여러 개면 모두 제시한다. 말없이 하나 고르지 않는다.
- 더 단순한 방법이 있으면 말한다. 필요하면 반대 의견을 낸다.
- 불명확하면 멈춘다. 무엇이 헷갈리는지 짚고 묻는다.

## 2. 단순함 우선

**문제를 푸는 최소 코드. 투기적인 것 금지.**

- 요청하지 않은 기능 추가 금지.
- 일회성 코드에 추상화 금지.
- 요청 없은 "유연성"·"설정 가능성" 금지.
- 일어날 수 없는 시나리오에 대한 에러 처리 금지.
- 200줄 썼는데 50줄로 가능하면 다시 쓴다.

자문: "시니어 엔지니어가 이거 과하다고 할까?" 그렇다면 단순화한다.

## 3. 외과적 변경

**꼭 필요한 것만 건드린다. 내가 만든 것만 치운다.**

기존 코드를 편집할 때:
- 인접한 코드·주석·포매팅을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩터링하지 않는다.
- 내 스타일과 달라도 기존 스타일을 따른다.
- 관련 없은 죽은 코드를 발견하면 언급만 한다. 지우지 않는다.

내 변경이 고아를 만들면:
- 내 변경으로 안 쓰이게 된 import·변수·함수는 제거한다.
- 기존부터 죽어 있던 코드는 요청 없이 지우지 않는다.

기준: 바뀐 모든 줄이 사용자 요청으로 직접 추적돼야 한다.

## 4. 목표 주도 실행

**성공 기준을 정의한다. 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 바꾼다:
- "검증 추가" → "잘못된 입력에 대한 테스트를 쓰고, 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 쓰고, 통과시킨다"
- "X 리팩터링" → "전후로 테스트가 통과하는지 보장한다"

여러 단계 작업은 간단한 계획을 명시한다:
```
1. [단계] → 검증: [체크]
2. [단계] → 검증: [체크]
3. [단계] → 검증: [체크]
```

강한 성공 기준은 독립적으로 반복하게 해준다. 약한 기준("되게 해줘")은 계속 확인이 필요하다.

## 5. 한국어 출력에서 종결 콜론 금지

**한국어 문장은 콜론(:)이 아니라 마침표로 끝낸다.**

사용자가 한국어로 쓰면 출력도 한국어다:
- 다음 줄이 리스트·예시여도 문장을 `:`로 끝내지 않는다.
- 영어 문서로 학습한 LLM은 한국어에 콜론 습관을 흘린다. 잡아낸다.
- 기준: 모든 한국어 문장 종결부는 `.`, `?`, `!` 중 하나여야 한다. `:`는 아니다.
- 코드·키값 쌍·라벨 안의 콜론은 괜찮다. 문장 종결로서가 아니다.

## 6. 파일 헤더 주석은 한국어로

**새 소스 파일 첫 줄: 그 역할을 한 줄 한국어 주석으로 적는다.**

새 파일을 만들 때:
- TypeScript/JavaScript: `// 사용자 인증 상태를 관리하는 Context Provider`
- Python: `# KIS API 호출을 비동기로 처리하는 클라이언트`
- SQL: `-- 일별 집계 결과를 저장하는 머터리얼라이즈드 뷰`
- 필수 지시문(`'use client'`, `'use server'`, shebang) 바로 아래에 둔다.
- 설정 파일(`*.config.ts`, `package.json` 등)은 건너뛴다.

이유: 에이전트는 코드베이스 전체가 아니라 선택적으로 파일을 읽는다. 한 줄 한국어 헤더가 즉시 맥락을 주어, 다음 세션(사람이든 에이전트든)이 전체를 다시 읽지 않고 탐색할 수 있다.

## 7. 계획 + 체크리스트 + 컨텍스트 노트

**사소하지 않은 작업 전에 세 산출물을 만든다. 이것 없이 코딩 시작 금지.**

- **계획** — 무엇을 왜 만드는가.
- **체크리스트**(`checklist.md`) — 구체적 작업을 체크박스로. 진행하며 체크.
- **컨텍스트 노트**(`context-notes.md`) — 작업 중 내린 결정과 그 근거. 계속 덧붙임.

사용자가 계획만 주고 코딩 시작을 요청하면, 멈추고 묻는다: "체크리스트와 컨텍스트 노트를 먼저 만들까요?" 다음 세션(내 것이든 남의 것이든)은 모든 결정을 다시 도출하지 않고 이어가려면 노트가 필요하다.

## 8. 완료 표시 전에 테스트 실행

**코드를 건드렸으면 "완료" 말하기 전에 테스트를 돌린다.**

- `npm test`, `pytest`, `cargo test` 등 프로젝트가 쓰는 것을 돌린다.
- 통과하면 결과 보고. 실패하면 고치고 다시 돌린다.
- 테스트 셋업이 없으면 최소한 빌드/컴파일되는지 확인한다.
- 사용자가 "돼", "완료", "다 됐어" 신호를 주기 전에 능동적으로 돌린다.

LLM이 가장 자주 건너뛰는 단계다. 타협 불가로 취급한다.

## 9. 시맨틱 커밋

**하나의 논리적 변경이 끝나면 커밋한다. 사용자가 요청할 때까지 기다리지 않는다.**

- 기준: "이 커밋을 한 문장으로 설명할 수 있나?" 그렇다면 커밋. 아니면 변경이 섞인 것 — 나눈다.
- 좋음: "auth 미들웨어 추가". 나쁨: "auth 추가하고 UI도 고치고 버그도 수정" (3개로 분리).
- 관련 없는 편집 20개를 쌓아 개별 롤백 능력을 잃지 않는다.
- 커밋을 위한 커밋 금지 — 의미 있는 단위만.

비고: 1인 프로토타입·일회성 스크립트는 속도를 늦추면 느슨하게 묶어 커밋해도 된다. 핵심은 되돌릴 수 있음이지 격식이 아니다.
(이 레포 한정: git 미사용이며, 커밋/푸시는 사용자가 요청할 때만.)

## 10. 에러를 읽어라, 추측하지 마라

**실제 에러/로그 줄을 읽는다. 기억에서 패턴 매칭하지 않는다.**

뭔가 실패하면:
- 전체 에러 메시지와 스택 트레이스를 읽는다.
- 가정이 아니라 실제 로그 출력을 확인한다.
- 원인 확인 전에 "흔한 수정"을 적용하지 않는다.
- 불명확하면 print/log를 넣어 상태를 확인한 뒤 고친다.

"테스트 실행" 다음으로 LLM이 가장 자주 건너뛰는 단계다. 에러 키워드로 추측해 최근 패턴 수정을 적용한다. 그렇게 한 줄 버그가 세 파일 리팩터링이 된다.
